Chapter 8 · Part III — Async

Async SQLAlchemy

Everything you've learned (engines, sessions, select(), loader options) survives intact; you mostly add await. What changes is one deep rule: no implicit IO, ever. This chapter explains where that rule comes from and catalogs every place it bites.

How it works: greenlets all the way down

SQLAlchemy's internals, the unit of work, the compiler, a decade of battle-tested machinery, are synchronous code. Rather than maintain a parallel async rewrite, SQLAlchemy runs that sync machinery inside a greenlet (a coroutine-like construct that can be paused from anywhere), and when the code reaches the point of actual database IO, it context-switches out to the event loop, awaits the truly-async driver (asyncpg), and switches back. Your await session.execute(...) hops into a greenlet, runs the familiar sync path, and surfaces each driver call as a real awaitable.

This buys perfect API symmetry with a catch: the escape hatch only exists inside that greenlet bridge. Any code path that tries to do database IO outside it, like a lazy load triggered by innocent attribute access, cannot await anything, and raises the exception this chapter is really about:

⏭ 2.1 watch

In 2.1 greenlet stops installing automatically. You opt in with pip install "sqlalchemy[asyncio]". Recent 2.0.x releases already skip it on some platforms (this course's own pyproject.toml hit exactly that and uses the extra). Write the extra into your dependencies today and the upgrade is a non-event.

The async vocabulary

Note the precision of the awaits: session.add() isn't awaited (Chapter 5 taught you it's pure bookkeeping), while commit(), scalars(), and refresh() are. The async API is an honest map of where IO actually happens, which is why learning the sync rhythm first was the right order. The pool is an AsyncAdaptedQueuePool automatically; all of Chapter 2's sizing math still applies.

The gotcha catalog

examples/ch08/02_missing_greenlet.py triggers each of these on purpose. Run it; the errors are better teachers in person.

1. Lazy loads

Notice the philosophical inversion: in sync land, the N+1 problem (Chapter 7) was silent. Everything worked, slowly. Async SQLAlchemy makes hidden IO impossible instead. It's the raiseload("*") discipline, enforced by physics. Annoying on day one; the correct default forever after.

2. expire_on_commit again, and now it matters

Chapter 5 planted this flag: after commit(), every attribute is expired and the next access reloads. Reload means IO, and IO means a greenlet error. So touching obj.anything after an async commit, even a plain column, explodes:

Every credible async example, including SQLAlchemy's own docs, sets this. Treat it as part of the async_sessionmaker incantation.

3. One session per task, one engine per event loop

An AsyncSession is a single-task creature. For asyncio.gather() fan-out, each task opens its own session from the shared factory: examples/ch08/03_concurrency.py runs 8 workers over a 5-connection pool to show the pattern, and that the pool queues gracefully. An AsyncEngine is bound to the event loop its connections were created on. Don't create it at import time in one loop and use it in another (a classic pytest bug, see Chapter 10); call await engine.dispose() before crossing loops, and always at shutdown, or garbage collection prints RuntimeError: Event loop is closed at exit.

🏭 In production

asyncpg plus pgbouncer in transaction mode is a notorious pairing: asyncpg caches prepared statements per connection, but pgbouncer hands "your" connection to someone else between transactions, and the result is DuplicatePreparedStatementError at 2am. If you must run through pgbouncer, disable both caches in connect_args: statement_cache_size=0 and prepared_statement_cache_size=0 (guides that mention only the first reduce the error rate without eliminating it). Cleaner: point SQLAlchemy at PostgreSQL directly with normal pooling, skipping NullPool, and let pgbouncer serve the clients that need it.

Do you even need async?

An honest aside before Part IV: async shines when your service holds many concurrent requests that spend their time waiting, on the database, on other APIs. A background job, a CLI, a low-traffic admin app gains nothing but complexity. Sync SQLAlchemy with a properly sized pool is boring, fast, and immune to every gotcha in this chapter. Choose async because your workload is concurrent-IO-bound, not because it's modern. (FastAPI handles sync endpoints in a threadpool just fine.)
