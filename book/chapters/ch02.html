Chapter 2 · Part I — Foundations

Engines & connections

The Engine looks boring. One call to create_engine() and you never think about it again. That's exactly why it causes so many production incidents. This chapter makes you the person who understands the pool.

One Engine, for the life of the process

An Engine is three things bolted together: a dialect (how to speak PostgreSQL), a connection pool (a bag of live TCP connections), and a URL saying where to connect. It is expensive to build, cheap to use, and completely thread-safe:

The rule: one Engine per database, per process, created once at startup. Creating engines per request is the classic beginner-to-intermediate mistake. Every one starts a cold pool, and your database watches connection counts sawtooth while your latency graphs suffer.

Also note what doesn't happen at construction: nothing. No network, no validation. The first real connection is made on first use, so a typo'd password surfaces at the first query, not at startup. If you want fail-fast behavior, do a SELECT 1 during boot.

🏭 In production

Don't build URLs with f-strings. A password containing @, / or % will shred your URL in a way you'll debug for an hour. URL.create(...) takes the parts as keyword arguments and handles escaping. It also keeps the password out of repr(), since SQLAlchemy masks it when the URL is printed or logged.

Connections, and the rollback surprise

engine.connect() checks a connection out of the pool and hands you a Connection. The with block returns it. Here is the part that ambushes almost everyone at least once (examples/ch02/01_connect_vs_begin.py):

Modern SQLAlchemy has no library-level autocommit. A connection autobegins a transaction on first use, and when that connection goes back to the pool, the pool issues a rollback to guarantee the next borrower gets a clean connection. Uncommitted work evaporates, silently and by design.

engine.begin() is the one you'll reach for most. It's the shape of an atomic business operation. The chapter challenge makes you move money with it, using a test that tries to make money vanish.

🔬 Under the hood

"No autocommit" is SQLAlchemy policy, not a driver limitation. The DBAPI spec itself says connections open in transaction mode; 1.x SQLAlchemy papered over it with "autocommit" magic that guessed when you meant to commit (it looked for INSERT/UPDATE/DELETE in your SQL, really). 2.0 removed the guessing. If you genuinely want driver-level autocommit, say for VACUUM or CREATE DATABASE which can't run in a transaction, it's explicit: engine.execution_options(isolation_level="AUTOCOMMIT").

The pool: your app's most important queue

Opening a PostgreSQL connection costs a TCP handshake, TLS negotiation, authentication, and a backend process fork on the server: tens of milliseconds and real server memory. Doing that per query would be ruinous, so the Engine keeps a QueuePool of live connections and lends them out. The parameters that define its behavior:

ParameterDefaultMeaning
pool_size5Connections kept open persistently
max_overflow10Extra connections allowed under burst; closed when returned
pool_timeout30 sHow long a checkout waits for a free connection before raising
pool_recycle-1 (off)Retire connections older than N seconds
pool_pre_pingFalseTest each connection with a ping at checkout

Do the arithmetic: defaults allow 5 + 10 = 15 concurrent checkouts. The 16th request waits up to 30 seconds, then raises. You can watch this happen in fast-forward. The demo uses a pool of 2+1 and a 1.5 s timeout (examples/ch02/02_pool_anatomy.py):

⚠ Gotcha

When QueuePool limit ... reached shows up in production, the reflex is to raise pool_size. Resist it. In the wild this error is almost always a connection leak: some code path checks out a connection (or Session) and never closes it, often only on an error branch. A bigger pool just delays the same failure and adds load to the database. Hunt the leak: every connect()/Session() belongs in a with block, and engine.pool.status() tells you how many connections are checked out right now.

Connections die. Plan for it.

A pooled connection can sit idle for minutes, plenty of time for a firewall, a load balancer, a database restart, or a cloud provider's NAT gateway to quietly kill the TCP stream. The next borrower gets a dead connection and a miserable stack trace. Two defenses:

  • pool_pre_ping=True: pessimistic and simple. On every checkout the pool sends a lightweight ping; if it fails, the connection (and any others older than it) is replaced before your query runs. It costs one round-trip per checkout. For almost every service the right answer is to turn it on.
  • pool_recycle=1800: retire connections before the infrastructure does. Thirty minutes is a sane figure for most networks (the famous motivator is MySQL's default 8-hour wait_timeout, but AWS NAT gateways at 350 s idle are the modern culprit).

examples/ch02/03_production_engine.py wires both, plus pool event listeners so you can watch checkouts, reuse, and replacement of an invalidated connection happen live.

🏭 In production

Two pools don't stack. If your infrastructure uses pgbouncer in transaction-pooling mode, SQLAlchemy's QueuePool holding long-lived connections works against it and produces confusing failures. The standard advice is to give SQLAlchemy poolclass=NullPool and let pgbouncer own pooling, or skip pgbouncer until you have the scale that demands it. (There's an extra asyncpg-specific trap here; chapter 8 covers it.)

Pools don't survive fork. Gunicorn's prefork workers and multiprocessing both copy the parent's file descriptors, and two processes sharing one TCP connection is corruption roulette. Create the engine after fork, or call engine.dispose(close=False) in each child's initializer.
