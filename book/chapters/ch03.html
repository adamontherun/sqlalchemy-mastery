Chapter 3 · Part I — Foundations

Querying with Core

The SQL Expression Language is the beating heart of SQLAlchemy. The ORM is a layer on top of it, not beside it. Learn it well here and Part II becomes easy, because the querying never changes.

Schema as data: MetaData and Table

Before you can query tables, SQLAlchemy needs Python objects that describe them. A Table is exactly that: a description, collected into a MetaData registry:

Because it's data, you can do data things with it: compile it to DDL and print it, walk its columns, hand it to Alembic for migrations, or call metadata.create_all(engine), which emits CREATE TABLE for anything missing and never touches tables that already exist. (It also never alters anything: schema evolution is Alembic's job, chapter 9.)

In Part II you'll define schemas with ORM classes instead, but every ORM model generates one of these Table objects underneath (it lives at Model.__table__). Nothing you learn here is thrown away.

insert(): the executemany trick

🔬 Under the hood

That list-of-dicts form triggers insertmanyvalues, a 2.0 feature: SQLAlchemy rewrites the INSERT to carry many VALUES (...), (...), ... tuples per statement, even combined with RETURNING, which classic DBAPI executemany can't do. That's how the ORM flushes thousands of new objects fast, too.

⚠ Gotcha

Result.rowcount is reliable for single-statement UPDATE and DELETE, but for batched inserts many drivers report -1. If you need an exact count from a bulk insert, add .returning(table.c.id) and count the rows that come back. The chapter challenge makes sure you meet this one in a controlled environment.

select(): composition over concatenation

Every clause you know from SQL has a method, and every method returns a new statement. They're immutable, so it's safe to build them up across branches and helper functions:

Note what you did not do: interpolate a single value into a string. Every Python value became a bound parameter automatically. For richer conditions there are or_(), and_(), not_(), Column.in_(), .like(), .between(), .is_(None) (SQL's IS NULL; == None works too but linters hate it), and when the operator you want doesn't exist: column.op("<->")(value) compiles any operator verbatim.

Reading results: the Result API

conn.execute() returns a Result; iterating yields Row objects that behave like named tuples: row.title, row[0], and unpacking all work. The methods you'll actually use:

CallReturnsReach for it when
.all()list[Row]You want every row
.first()Row | NoneJust the first row, if any
.one()RowExactly one row must exist; raises otherwise
.scalars()column valuesOne-column queries; skip the Row wrapper
.scalar_one()a single valueOne row, one column: counts, ids, EXISTS
.mappings()dict-like rowsYou're serializing to JSON anyway

The .one() family is quietly one of SQLAlchemy's best habits: it turns "impossible" data states into immediate, loud errors instead of a None that detonates three functions later.

update() and delete(): set-based, like SQL intended

⚠ Gotcha

See that Decimal("0.9")? Multiply a NUMERIC column by a Python float and PostgreSQL silently promotes the expression to double precision. At that point round(..., 2) fails with function round(double precision, integer) does not exist. Keep money in Numeric/Decimal end to end; the moment a float sneaks in, PostgreSQL will tell you. This error message is a rite of passage.

Joins and aggregates

join_from(left, right) makes the FROM side explicit; plain .join(right) works when it's unambiguous. The join condition comes from the ForeignKey, or you can pass one explicitly as a second argument. And func is an open namespace: func.count, func.avg, func.jsonb_agg, func.anything_your_db_has. SQLAlchemy compiles the name verbatim and lets the database be the judge.

text(): hand-written SQL is not cheating

SQLAlchemy's own docs are emphatic on this: dropping to raw SQL for a gnarly report, a vendor-specific feature, or a query the planner likes better is normal, supported practice, with :named bound parameters and the full Result API. The library's promise was never "no SQL"; it was "no string-glued SQL."
