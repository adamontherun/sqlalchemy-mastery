Chapter 10 · Part IV — Production

Testing

Test against the real database. Mocking SQLAlchemy tests your mocks, and SQLite-in-memory tests a different database than you ship. The trick that makes real-Postgres tests fast and independent is one nested transaction.

Why not mock? Why not SQLite?

Your SQLAlchemy code's job is to produce correct SQL for your actual database. Mock the session and you've verified that your code calls the methods you mocked, which is a tautology. Swap in SQLite "for speed" and you're testing a database with different types, different constraint enforcement, no ON CONFLICT semantics you rely on, and none of the PostgreSQL features (JSONB, enums, RETURNING nuances) this course has been using. You have PostgreSQL 17 in Docker one compose up away. Test against it.

The objection is speed and isolation: tests that share a database dirty it for each other, and re-creating schemas per test is slow. Both are solved by the same pattern.

The rollback-per-test pattern

Open a connection and a transaction the test can't see. Bind the test's Session to that connection with join_transaction_mode="create_savepoint": every commit() the code under test performs becomes a savepoint release inside the outer transaction. Teardown rolls the outer transaction back, and everything, including "committed" work, vanishes. From examples/ch10/conftest.py:

🔬 Under the hood

Why a savepoint? If the Session simply shared the outer transaction, the first session.commit() in app code would commit it for real, and teardown would have nothing to roll back. create_savepoint mode makes the Session treat the outer transaction as untouchable and conduct all its "transactions" as nested SAVEPOINT/RELEASE, which PostgreSQL supports natively and cheaply.

The async flavor

Same idea with AsyncSession. See the async fixture in the same conftest. Two async-specific rules from Chapter 8 apply doubly in tests: pytest-asyncio can give each test its own event loop, and an async engine must not hop loops (the symptom is baffling RuntimeError: ... attached to a different loop / Event loop is closed flakiness), so create the engine inside the fixture and dispose it at the end. And asyncio_mode = "auto" in pyproject.toml (this repo sets it) lets you write bare async def test_... functions without decorating each one.

🏭 In production

Two upgrades teams commonly add: run the suite's schema from Alembic migrations instead of create_all once migrations exist, so your migration chain is tested on every CI run for free; and add raiseload("*") as a default loader option in test fixtures, so any relationship access that would lazy-load in production fails loudly in CI first (chapter 7's N+1s, caught mechanically). For CI without a shared database, testcontainers starts a throwaway Postgres per run with the same fixtures and zero infrastructure.

⚡ Try it

No local setup? Open this chapter in GitHub Codespaces — Postgres is already running there.

Run the demo suite and read its output: uv run pytest examples/ch10 -v: five tests proving commit-then-vanish isolation, sync and async.

Challenge: uv run pytest challenges/ch10. Build the harness yourself as a reusable class. The meta-tests commit data, then check from a separate connection that nothing leaked, and run your harness against a one-connection pool to prove it doesn't leak connections either.
