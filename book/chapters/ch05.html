Chapter 5 · Part II — The ORM

The Session

The Session is the ORM's brain: a workspace that watches your objects, computes the SQL you didn't write, and guarantees one Python object per database row. Understand its rhythm of accumulate, flush, commit, and the ORM stops feeling magical and starts feeling inevitable.

The unit of work: nothing happens until it has to

When you session.add() an object, no SQL runs. When you change an attribute of a loaded object, no SQL runs. The Session accumulates intent, and only converts it to INSERT/UPDATE/DELETE at a flush, which happens automatically right before any query (so your queries see your own pending changes), and unconditionally at commit. Run examples/ch05/01_unit_of_work.py with its SQL echo and watch the timing:

This is why the ORM can batch three INSERTs into one insertmanyvalues statement, order operations to satisfy foreign keys, and skip UPDATEs for objects that didn't actually change. It sees the whole picture before writing anything.

The other pillar is the identity map: within one session, a row has exactly one Python object. Load "user #7" through two different queries and you get the same object, is-identical. No stale copies of the same row disagree with each other; that class of bug simply doesn't exist inside a session.

⚠ Gotcha

The identity map is not a query cache. Only session.get(User, 7) (primary-key lookup) can skip SQL. A select().where(User.email == ...) always hits the database; it just reuses the existing object when rows come back. If you expected caching, you'll see more SQL than you thought; if you expected fresh data mid-transaction, note that attribute values of already-loaded objects are not overwritten by later queries.

The four lives of an object

StateMeaningHow it got there
transientPlain object; no session, no rowUser(...)
pendingQueued for INSERT at next flushsession.add()
persistentHas a row; watched for changesflush, or loaded by a query
detachedHas a row; nobody watchingsession.close() / eviction

examples/ch05/02_object_states.py walks one object through all four (plus merge() to re-attach a detached object; useful, but if you're merging a lot, your session boundaries are probably wrong). The state lives at inspect(obj) when you need to debug.

flush ≠ commit

Flush writes pending changes to the database inside the open transaction. Generated primary keys appear on your objects, but nothing is durable yet; a rollback still undoes it. Commit flushes, then issues COMMIT. That part is durable. And then, by default, everything expires.

expire_on_commit: the default that explains two chapters

After commit(), the session marks every attribute of every object expired: the next attribute access silently runs a SELECT to reload fresh data. Watch it happen (examples/ch05/03_expire_on_commit.py):

Inside a plain sync script this is merely surprising. But that attribute-access-that-does-IO is precisely what cannot happen in two important places: async code (no implicit IO allowed; MissingGreenlet, Chapter 8) and web handlers that touch objects after the session closed (DetachedInstanceError, Chapter 12). Both cures start with Session(engine, expire_on_commit=False), which keeps your loaded state and accepts it may be a hair stale after commit. Remember this section; we will point back at it twice.

Session lifecycle: short, scoped, external

The official guidance compresses to three rules. One session per logical operation: a request, a job, a CLI command, measured in minutes, not hours, and never global. The function that uses a session shouldn't create it; sessions are passed in (or dependency-injected, Chapter 12), and the docs literally label embedded session management "the wrong way to do it." And one session, one thread or task: the Session is not thread-safe, by design. The engine is; share that instead.

🏭 In production

A session holds a database connection from its first query until commit/rollback/close. A "convenient" long-lived session (module global, class attribute, "we'll just reuse it") is therefore also a connection leak (remember the Chapter 2 pool math), an ever-growing identity map (a memory leak), and a long-running transaction (table bloat and lock contention in PostgreSQL). The chapter challenge runs your code against a pool of exactly one connection: leak a single session and the tests deadlock.
