Chapter 11 · Part IV — Production

Performance

Slow SQLAlchemy is almost never SQLAlchemy being slow. It's your code asking for the wrong shape of work: too many round-trips, too many transactions, too many objects. Every fix in this chapter is about changing the shape.

Round-trips and transactions dominate everything

The numbers from examples/ch11/01_bulk_inserts.py, inserting 5,000 rows on this course's own setup:

A 100× difference, and no index or hardware was involved. The per-row version pays a full transaction, including the WAL fsync, five thousand times. The lesson generalizes: batch your work, commit once per logical operation, and if the rows are just data (no identity, no behavior), skip building ORM objects entirely and hand insert() a list of dicts.

Finding the waste: read what you emit

Turn on create_engine(echo=True) in development, then actually read it. N+1 patterns (Chapter 7) are unmissable once seen: the same SELECT with different parameters, over and over. The echo also shows [cached since ...] on repeated statement shapes, 2.0's compiled-statement cache at work. It's why building the same select() in a loop is cheaper than it looks, and why wildly dynamic query construction is more expensive than it looks. In production, use the logging module (sqlalchemy.engine logger) or better, the two-event slow-query logger from examples/ch11/03_pool_monitoring.py: before_cursor_execute stamps a start time and after_cursor_execute reports anything over threshold. And export engine.pool.status() numbers as metrics: a checked-out count pinned at pool max is the leak signature from Chapter 2, visible before the TimeoutErrors start.

Streaming: constant memory for unbounded rows

.all() materializes every row and every ORM object at once. For exports, ETL, and reports, stream instead. examples/ch11/02_streaming.py measures 50,000 rows:

yield_per combines a server-side cursor (PostgreSQL holds the result, sending batches) with batched ORM construction. Two caveats: the connection stays busy until you finish iterating, and it's incompatible with joined or subquery collection eager loading (both need the full parent set up front). selectinload is the documented way around that, since it issues its own follow-up query per batch instead of joining against the whole result.

⚠ Gotcha

Beware the accidental object count: session.scalars(select(Order)) .all() on 100k rows builds 100k tracked Python objects. The identity map holds them all, change-tracking watches them all, and your memory graph looks like a cliff. If you're not going to mutate them, you didn't need tracked objects. This single habit, selecting entities when columns would do, is the most common ORM performance sin after N+1.

🏭 In production

When a query is slow in the database (not in Python), no SQLAlchemy knob helps. Get the real SQL from echo, run EXPLAIN ANALYZE on it, and fix the index or the query shape. SQLAlchemy's contribution is that the SQL it emits is plain, predictable, and copy-pasteable into psql, so use that. PostgreSQL's pg_stat_statements extension is the server-side complement to the client-side logging above.
