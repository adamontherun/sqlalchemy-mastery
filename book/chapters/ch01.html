Chapter 1 · Part I — Foundations

The lay of the land

SQLAlchemy is not an ORM. It contains one, and a very good one, but thinking of it as "the Python ORM" is how people end up fighting it. This chapter builds the mental model everything else hangs off.

A toolkit, arranged in layers

You already speak SQL, which is a real advantage here, because SQLAlchemy's deepest design commitment is that it never hides SQL from you. Its own docs call the philosophy "SQLAlchemy considers the database to be a relational algebra engine, not just a collection of tables." Every layer of the library is a thin, inspectable translation of things you already know.

The layers, from the metal up:

  1. The DBAPI driver. A third-party package that actually talks the wire protocol. For PostgreSQL in this course that's psycopg (sync) and asyncpg (async). SQLAlchemy never sends a byte itself.
  2. The dialect. SQLAlchemy's adapter for one database and driver combination. It knows that PostgreSQL spells string concatenation ||, how RETURNING works, how to reflect a schema, and a hundred other quirks.
  3. The Engine and its connection pool. The factory for database connections and the pool that recycles them, one per database, per process, for the life of your application. Chapter 2 is all about this.
  4. Core, the SQL Expression Language. Python objects that are SQL: select(), insert(), Table, Column. Composable, safely parameterized, database-agnostic where possible and database-specific where you want it.
  5. The ORM. Built entirely on top of Core. It adds classes mapped to tables, the Session's unit of work, relationships, and identity-mapped objects. It compiles down to Core; Core compiles down to SQL strings; the driver ships them.
🔬 Under the hood

When you write select(User).where(User.name == "adam"), the == doesn't compare anything. User.name is an instrumented attribute that overloads == to return a BinaryExpression, a node in a SQL syntax tree. The tree only becomes the string SELECT ... WHERE users.name = %(name_1)s when the dialect compiles it at execution time. You are writing SQL with Python operator overloading, not calling an API that generates SQL.

Core or ORM: which do you use?

Both, usually. For a modern application, define your schema and everyday persistence with the ORM, and drop into Core expressions freely for reporting queries, bulk operations, and anything where you want rows rather than objects. Since 2.0, this is one skill, not two. The same select() construct works against tables and mapped classes alike, executed by a Connection (Core) or a Session (ORM).

You wantReach for
Domain objects with identity and relationshipsORM · Session
Plain rows, aggregates, reportsCore · Connection, or ORM returning rows
Bulk load of a million rowsCore insert() with parameter lists
One weird vendor-specific statementtext(): hand-written SQL is a first-class citizen

Why old tutorials will mislead you

SQLAlchemy is over twenty years old, and the internet is a sedimentary record of every era. In 2023 the project landed 2.0, the largest cleanup in its history, and the version this book targets (2.0.51 as of writing). If a tutorial shows any of the following, it's teaching you the 1.x world:

  • session.query(User).filter_by(...), the legacy Query API. Modern code writes session.execute(select(User).where(...)) or the session.scalars(...) shorthand.
  • Base = declarative_base(), replaced by class Base(DeclarativeBase), which type checkers understand.
  • Column("name", String(50)) in models, replaced by Mapped[str] annotations with mapped_column(), giving you real static typing end to end.
  • engine.execute(...) or bare string execution. Implicit "autocommit" connections are gone; transactions are always explicit now.

None of the old spellings make you faster; they just make your code look like an archaeological dig. The 2.0 API is smaller, typed, and identical in shape between sync and async, which will pay off enormously in Chapter 8.

⏭ 2.1 watch

SQLAlchemy 2.1 (in beta as this book is written) keeps the 2.0 API intact. Notable changes: greenlet stops installing automatically (async users install sqlalchemy[asyncio]), Python 3.10 becomes the floor, and typed results drop the Tuple wrapper: Select[Tuple[int, str]] becomes Select[int, str]. Everything you learn here carries over.

Read it as a tour of the layers:

  • The URL postgresql+psycopg://… names the dialect (postgresql) and the driver (psycopg) explicitly. Swap +psycopg for +asyncpg and, with the async engine, the rest of your SQL knowledge transfers untouched.
  • create_engine() builds the Engine lazily. No connection happens until first use.
  • engine.connect() checks a connection out of the pool; the with block returns it. Forgetting this is the #1 way to exhaust a pool in production. Chapter 2 has the receipts.
  • text() wraps literal SQL, and :a/:b are bound parameters, never f-strings. The driver sends the values separately from the statement, so SQL injection is structurally impossible here.
  • .scalar_one() is the Result API insisting on exactly one row, one column. It raises if the shape surprises you. Loud failure is a recurring SQLAlchemy virtue.
⚠ Gotcha

A plain engine.connect() block rolls back anything you didn't commit. Run an INSERT inside it, exit the block, and your row silently vanishes: the pool resets connections with a rollback on return. This bites almost everyone once. Chapter 2 covers the two transaction patterns (connect() plus explicit commit(), versus begin()) that make intent explicit.
