Chapter 12 · Part IV — Production

FastAPI integration

The finale assembles everything: one engine per app (ch2), one session per request (ch5), eager loading for serialization (ch7), async rules honored (ch8). A complete working service lives in examples/ch12/bookmarks_api.py. This chapter walks its four load-bearing walls.

Wall 1: the engine lives in lifespan

Built once at startup, disposed once at shutdown: the Chapter 2 rule in FastAPI clothing. Creating the engine at import time works until the day something imports your module from a different event loop (a test runner, a worker); lifespan makes the lifecycle explicit and loop-correct. Schema creation in lifespan is demo convenience; real deployments run alembic upgrade head as a release step (Chapter 9).

Wall 2: session-per-request, as a dependency

This is Chapter 5's "sessions are created at the operation boundary and passed in," where the operation is an HTTP request and the passing is Depends. FastAPI caches dependencies per request, so every layer that asks for SessionDep (handler, service, repository) shares one session and therefore one transaction. Commit happens in exactly one place; endpoints never call commit() themselves (they may flush() to get generated ids; flush is not commit, Chapter 5).

Wall 3: serialization happens after the session closes

The subtle one. Your handler returns an ORM object; FastAPI/Pydantic serializes it after the dependency generator finished, session closed, transaction committed. Two Chapter 8 traps are armed at exactly that moment. If attributes expired at commit, a reload attempt outside the session raises MissingGreenlet or DetachedInstanceError, which is why expire_on_commit=False in the factory is non-negotiable in this architecture. Wall 4: errors roll back, and the app keeps working

A unique-constraint violation shouldn't 500. The pattern: attempt the flush(), catch IntegrityError, translate to a semantic status code. The dependency's except branch handles the rollback; the failed request leaves no debris, and the next request gets a clean session from the pool. The chapter challenge tests precisely this: a duplicate POST must 409 and the app must keep answering afterwards.

🏭 In production

Sizing note that unifies chapters 2 and 8: each uvicorn worker process has its own engine and pool. Four workers × (pool_size=10 + max_overflow) can mean 60+ connections, so check your PostgreSQL max_connections (default 100) before the load test does. And the classic FastAPI mistake worth repeating: a def endpoint (no async) doing sync SQLAlchemy is fine, since it runs in a threadpool. An async def endpoint doing sync SQLAlchemy blocks the event loop for everyone. Pick one stack per service and be consistent.

Where you are now

You can read a SQLAlchemy stack trace and know which layer produced it. You know why the pool times out, why the Session flushes when it does, what the async layer forbids and why, how schema change ships, and how to test all of it without flakes. Knowing the API wouldn't get you any of that; this is the understanding that debugging at 3am requires. The repo's notes/research.md collects the sources this course distilled if you want to go deeper. Go build something.

⚡ Try it

No local setup? Open this chapter in GitHub Codespaces — Postgres is already running there.

Run the service: uv run uvicorn bookmarks_api:app --app-dir examples/ch12 --port 8012, then poke it with the curl commands in the file's docstring, or open localhost:8012/docs and use the generated UI.

Final challenge: uv run pytest challenges/ch12. Build the integration yourself: factory, dependency, three endpoints, a 409 that rolls back cleanly, and eager loading that survives serialization. Everything this book taught, in forty lines of yours.
