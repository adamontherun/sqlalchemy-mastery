Chapter 9 · Part IV — Production

Migrations with Alembic

create_all() creates what's missing and alters nothing, fine for demos, useless once real data exists. Alembic (SQLAlchemy's sibling project) turns schema change into versioned, reviewable, reversible scripts.

The mental model

Your models are the desired state; the database is the current state; a migration is a diff with a direction: an upgrade() and (write it!) a downgrade(). Migrations form a chain of revisions, and the database remembers where it stands in a single-row table Alembic manages (alembic_version).

Setup is alembic init migrations plus two edits, both already made in the example project: point sqlalchemy.url at your database (in alembic.ini), and hand your metadata to autogenerate (in migrations/env.py): from models import Base; target_metadata = Base.metadata.

What autogenerate sees, and what it can't

Autogenerate compares metadata to the live schema. It reliably detects added/removed tables and columns, nullability changes, and named indexes/uniques/FKs. Its blind spots are where migration discipline lives:

ChangeWhat autogenerate doesWhat you must do
Renamed table or column Emits drop + add: your data would vanish Rewrite as op.alter_column(..., new_column_name=...) / op.rename_table(...)
Unnamed constraints Can't track them at all The ch04 naming convention: set it before your first migration
CHECK constraints Mostly not detected Add op.create_check_constraint() by hand
Server default changes Only with compare_server_default=True, imperfectly Verify by reading the diff
Type changes Detected (compare_type on by default since 1.12) Still check: some coercions need USING clauses on PG

The Alembic docs say it plainly: "autogenerate is not intended to be perfect". Generated scripts are a draft for review, not a release artifact. The rename case is the one that costs people data: a column rename reads as "drop column (with its contents), add empty column." Nothing in CI will catch that for you; reading the migration will.

⚠ Gotcha

Adding a NOT NULL column to a table with existing rows fails at upgrade time, since existing rows have no value for it. Ship it with server_default= (the example project's second migration adds word_count exactly this way), or migrate in two steps: add nullable, backfill, then set NOT NULL. On large, hot tables the two-step dance also avoids a long table lock.

Migration hygiene for teams

Review migrations like code: they are code, run exactly once, against the data you can least afford to lose. Write real downgrades; the discipline pays off the first time a deploy needs rolling back at speed (the chapter challenge enforces up-down-up cycling). Aim for one head per merge, since parallel branches create two "head" revisions that alembic merge heads reconciles; put alembic check (fails if models drifted from migrations) in CI and drift never reaches main. And run migrations at deploy, not import: a dedicated release step (alembic upgrade head), never "on app startup" in every replica at once, or they'll race for the lock.

🏭 In production

Async apps still migrate sync-ishly. Alembic's machinery is synchronous; for an asyncpg-only project, alembic init -t async generates an env.py that wraps the sync migration core in connection.run_sync(...) inside asyncio.run(...), the same greenlet bridge from Chapter 8, pointed the other way. A simpler alternative many teams choose is to keep a sync psycopg URL just for migrations. Either way use NullPool: a migration process needs one connection for ten seconds, not a pool.

⚡ Try it

No local setup? Open this chapter in GitHub Codespaces — Postgres is already running there.

Explore the project: cd examples/ch09/migration_project, then uv run alembic history, upgrade head, look at the tables, downgrade base. Then make your own third revision: add a column to models.py, autogenerate, and read the file before upgrading.

Challenge: uv run pytest challenges/ch09. Write upgrade()/downgrade() with op directives by hand against a live Operations object: named constraints, server defaults, and a stability test that cycles your migration twice.
